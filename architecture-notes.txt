CONTENT BROADCASTING SYSTEM - ARCHITECTURE NOTES
===================================================

1. AUTHENTICATION & RBAC FLOW
============================

JWT-Based Authentication:
- Users login with email/password → Server validates credentials → Issues JWT token
- Token contains: user id, email, role, expiration time
- Token sent in Authorization header: "Bearer <token>"
- Middleware validates token on protected routes

Role-Based Access Control (RBAC):
- Two roles: Principal, Teacher
- Principal permissions:
  * View all users and content
  * Approve/reject content
  * Delete any content
  * View system statistics
- Teacher permissions:
  * Upload content
  * View own content
  * Delete own content (only uploaded/rejected status)
  * Manage content scheduling

Authentication Middleware Flow:
1. Extract token from Authorization header
2. Verify JWT signature and expiration
3. Fetch user from database
4. Attach user to request object
5. Role-based middleware checks permissions

2. SUBJECT-BASED SYSTEM DESIGN
=============================

Content Organization:
- Content organized by subjects (Mathematics, Science, etc.)
- Each subject has independent rotation and scheduling
- Content slots represent subject categories
- Teachers can upload content for any subject

Data Flow:
- Content uploaded → Assigned to subject → Added to rotation queue
- Each subject maintains its own rotation schedule
- Public APIs can filter by subject or teacher

Benefits:
- Scalable subject management
- Independent subject rotations
- Easy content categorization
- Teacher-specific content organization

3. UPLOAD HANDLING APPROACH
===========================

File Upload Process:
1. Multer middleware handles multipart/form-data
2. File validation: type (JPG/PNG/GIF), size (max 10MB)
3. Unique filename generation: <name>-<timestamp>-<random>.<ext>
4. Local storage in /uploads directory
5. Database stores file metadata and URL

File Storage Strategy:
- Current: Local filesystem storage
- Scalable: Can be extended to cloud storage (S3)
- File cleanup: Automatic on content deletion
- Access: Static file serving via Express

Security Measures:
- File type validation using MIME types
- File size limits
- Unique filename generation prevents conflicts
- Upload rate limiting (20 uploads/hour)

4. APPROVAL WORKFLOW DESIGN
===========================

Content Lifecycle:
1. UPLOADED → PENDING (automatic transition)
2. PENDING → APPROVED/REJECTED (principal action)
3. APPROVED → Available for broadcasting
4. REJECTED → Visible to teacher with reason

Approval Process:
- Teachers upload content → Status: pending
- Principals view pending content queue
- Principal can approve or reject with reason
- Email notifications (future enhancement)
- Audit trail maintained (approved_by, approved_at)

Workflow Benefits:
- Content quality control
- Clear approval chain
- Audit capabilities
- Teacher feedback mechanism

5. SCHEDULING/ROTATION LOGIC (CRITICAL)
=======================================

Rotation Algorithm:
- Time-based rotation using current timestamp
- Each content item has rotation_duration (default 5 minutes)
- Cumulative duration calculation for position determination
- Subject-independent rotation cycles

Mathematical Logic:
1. Calculate total rotation cycle: Σ(content_duration)
2. Get current time in minutes: floor(current_timestamp / 60)
3. Calculate position: current_time % total_cycle_duration
4. Find content where position < cumulative_duration

Time Window Handling:
- Content respects start_time and end_time
- Outside time window = not active for rotation
- No time window = always active when approved
- Real-time validation on each request

Rotation Features:
- Automatic content switching
- Seamless transitions
- Time-based precision
- Subject isolation

6. DATABASE DESIGN DECISIONS
===========================

Table Structure:
- Users: Authentication and role management
- Content: Content metadata and status tracking
- Content_Slots: Subject categories
- Content_Schedule: Rotation configuration

Relationships:
- Users 1:N Content (teacher uploads)
- Content 1:1 Content_Schedule (rotation settings)
- Content_Slots 1:N Content_Schedule (subject organization)

Indexing Strategy:
- User email for quick authentication
- Content status for approval queries
- Content subject for subject-based queries
- Content uploaded_by for teacher content queries

Database Features:
- Automatic timestamp updates via triggers
- Foreign key constraints for data integrity
- Unique constraints to prevent duplicates
- JSON support for future extensibility
